### Part 1: The Vercel "Build-time" Variable Issue
**The Problem:**
In Next.js, all environment variables starting with `NEXT_PUBLIC_` are baked directly into your React code as plain text when the website is **built/compiled**, not when the website is running. 

In your original settings:
* The variable `NEXT_PUBLIC_API_URL` was configured in a way that was only available at *runtime*.
* When Vercel ran the build process (`next build`), the variable was empty, so the code fell back to connecting to `localhost:8000`.
* Additionally, the Vercel Dashboard had an outdated value from May 12 which was overriding the file settings.

**The Fix:**
We updated `vercel.json` to define `NEXT_PUBLIC_API_URL` inside the `build.env` block (which forces Vercel to make it available during the compilation phase) and updated it in the Vercel Dashboard.

---

### Part 2: The CORS Security Block (on Vercel Preview URLs)
**The Problem:**
When you pushed the changes, Vercel generated a unique preview deployment URL to test the app:
`https://lpu-code-aeeaq1mbc-ltprathams-projects.vercel.app`

For security, web browsers block websites from making API requests to different servers unless the API server explicitly permits that website's URL (this is called **CORS - Cross-Origin Resource Sharing**). 
Your FastAPI backend was configured to *only* allow requests from your main domain (`https://lpu-code-viz.vercel.app`). Because the preview URL had a random string (`aeeaq1mbc`), the backend blocked it.

**The Fix:**
I updated your FastAPI backend (`backend/main.py`) to use `allow_origin_regex`. Instead of just allowing the main URL, it now uses a pattern matching system (`https://lpu-code-.*\.vercel\.app`) that automatically allows **all** present and future preview URLs generated by Vercel.

---

### Part 3: The DNS Refusal (`ERR_NAME_NOT_RESOLVED`)
**The Problem:**
When your backend was fully configured, your frontend tried to request:
`https://lpu-codeviz-production.up.railway.app/explain`

Before your computer can load a website, it must convert the domain name (like `lpu-codeviz-production.up.railway.app`) into a numeric IP address. To do this, it asks a Domain Name System (DNS) server.
By default, your laptop asked your phone hotspot carrier's DNS server. However, your mobile carrier's DNS returned a `DNS operation refused` error. This usually happens because mobile carrier DNS servers have strict firewall policies, block newer domains, or have not synced their records yet.

**The Fix:**
By enabling **Secure DNS (DNS-over-HTTPS)** in Microsoft Edge, you instructed Edge to stop asking your mobile carrier's DNS server for directions. Instead, Edge now directly asks Google or Cloudflare's secure DNS servers, which resolved the Railway server's IP address instantly and bypassed the carrier's block.
